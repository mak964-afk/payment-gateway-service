# User Stories — Payment Gateway Service

Формат: **Connextra** (`As a … I want … so that …`) + **Acceptance Criteria** в
нотации **Gherkin** (Given–When–Then). Каждая история снабжена приоритетом
(MoSCoW) и оценкой в story points.

Акторы: **Студент**, **Онлайн-школа (`lms`)** (системный), **Эквайер** (внешняя
система), **ОФД** (внешняя система), **Финансовый оператор**, **Планировщик**
(системный, dunning/холд).

---

## US-01 — Оплата курса картой

**Приоритет:** Must · **SP:** 8

> **Как** студент,
> **я хочу** оплатить курс банковской картой за один шаг,
> **чтобы** быстро получить доступ к обучению и фискальный чек.

### Acceptance Criteria

```gherkin
Feature: Разовая оплата курса

  Background:
    Given студент выбрал курс стоимостью 4990 рублей
    And указал корректные платёжные данные на стороне эквайера (токен получен)

  Scenario: Успешная одностадийная оплата
    When lms инициирует оплату с idempotency-ключом "PAY-1"
    Then платёж переходит в статус "SUCCEEDED"
    And формируется фискальный чек прихода по 54-ФЗ
    And публикуется событие "payment.succeeded"
    And ответ возвращается не дольше чем за 900 мс по p95

  Scenario: Отказ эквайера (недостаточно средств)
    When эквайер отклоняет операцию с кодом "insufficient_funds"
    Then платёж переходит в статус "DECLINED"
    And студенту возвращается человекочитаемая причина отказа
    And фискальный чек НЕ формируется
    And доступ к курсу не выдаётся

  Scenario: Идемпотентность повторного запроса
    Given платёж с idempotency-ключом "PAY-1" уже успешно проведён
    When lms повторно отправляет запрос с тем же ключом "PAY-1"
    Then повторное списание не выполняется
    And возвращается результат ранее проведённого платежа
```

## US-02 — Подписка с рекуррентными платежами

**Приоритет:** Must · **SP:** 13

> **Как** студент,
> **я хочу** оформить подписку на библиотеку курсов с автопродлением,
> **чтобы** не оплачивать доступ вручную каждый месяц.

### Acceptance Criteria

```gherkin
Feature: Рекуррентные подписки

  Scenario: Первичная оплата и сохранение токена карты
    Given студент оформляет подписку "Библиотека PRO" за 1490 рублей в месяц
    When первый платёж проходит успешно
    Then создаётся подписка в статусе "ACTIVE"
    And токен карты сохраняется для будущих списаний (PAN не хранится)
    And дата следующего списания устанавливается через 1 месяц

  Scenario: Автопродление по расписанию
    Given активная подписка с наступившей датой списания
    When планировщик инициирует рекуррентное списание
    Then выполняется платёж по сохранённому токену без участия студента
    And при успехе дата следующего списания сдвигается на 1 месяц
    And формируется фискальный чек

  Scenario: Отмена подписки студентом
    Given активная подписка
    When студент отменяет подписку
    Then подписка переходит в статус "CANCELLED"
    And следующее рекуррентное списание не выполняется
    And доступ сохраняется до конца оплаченного периода
```

## US-03 — Двухстадийный платёж (холдирование и списание)

**Приоритет:** Must · **SP:** 13

> **Как** онлайн-школа,
> **я хочу** сначала захолдировать сумму, а списать её после подтверждения
> (например, после старта потока курса),
> **чтобы** не замораживать деньги студента раньше времени и легко отменять.

### Acceptance Criteria

```gherkin
Feature: Холдирование и списание (two-stage)

  Scenario: Холд с последующим списанием
    Given студент записался на курс с двухстадийной оплатой 4990 рублей
    When lms инициирует авторизацию (hold)
    Then сумма холдируется, платёж переходит в статус "AUTHORIZED"
    And фискальный чек ещё НЕ формируется
    When поток курса стартовал и lms инициирует capture
    Then сумма списывается, платёж переходит в "SUCCEEDED"
    And формируется фискальный чек прихода

  Scenario: Автоотмена просроченного холда
    Given платёж в статусе "AUTHORIZED" с холдом, которому более 7 дней
    When планировщик проверяет просроченные холды
    Then холд отменяется у эквайера
    And платёж переходит в статус "CANCELLED"
    And публикуется событие "payment.hold_expired"

  Scenario: Частичное списание холда
    Given захолдировано 4990 рублей
    When lms инициирует capture на 3990 рублей (скидка применена позже)
    Then списывается 3990 рублей
    And остаток холда (1000 рублей) освобождается
    And чек формируется на фактически списанную сумму
```

## US-04 — Фискализация чека по 54-ФЗ

**Приоритет:** Must · **SP:** 8

> **Как** онлайн-школа,
> **я хочу** автоматически формировать корректный фискальный чек по каждому
> платежу,
> **чтобы** соблюдать 54-ФЗ и не получать претензий от ФНС и студентов.

### Acceptance Criteria

```gherkin
Feature: Фискализация по 54-ФЗ

  Scenario: Чек прихода после успешной оплаты
    Given платёж "PAY-1" успешно проведён
    When PGS отправляет чек в ОФД
    Then чек содержит позиции, сумму, НДС, признак способа расчёта и email студента
    And чек получает фискальный признак (ФП) и номер
    And статус чека переходит в "REGISTERED" не позднее 5 минут после оплаты

  Scenario: Сбой ОФД — повтор фискализации
    Given ОФД недоступен при первой попытке отправки чека
    When срабатывает повтор с экспоненциальным backoff
    Then чек повторно отправляется до успеха или исчерпания попыток
    And платёж остаётся "SUCCEEDED" независимо от статуса чека
    And при исчерпании попыток заводится инцидент для оператора

  Scenario: Чек возврата прихода при refund
    Given по платежу "PAY-1" оформлен возврат
    When возврат проведён эквайером
    Then формируется фискальный чек "возврат прихода" на сумму возврата
```

## US-05 — Умные повторы неуспешного рекуррентного платежа (Dunning)

**Приоритет:** Should · **SP:** 13

> **Как** онлайн-школа,
> **я хочу** автоматически повторять неуспешные списания по подписке по
> разумному расписанию,
> **чтобы** снизить отвал подписчиков из-за временных отказов банка.

### Acceptance Criteria

```gherkin
Feature: Dunning — умные повторы

  Scenario: Повтор после временного отказа
    Given рекуррентное списание отклонено с кодом "try_again_later"
    When запускается dunning-стратегия
    Then платёж переходит в статус "RETRY_SCHEDULED"
    And следующая попытка планируется через 1 день (далее 3 и 5 дней)
    And подписка остаётся "ACTIVE" в течение grace-периода

  Scenario: Успешный повтор в рамках dunning
    Given запланирован повтор списания по подписке
    When повторная попытка проходит успешно
    Then платёж переходит в "SUCCEEDED"
    And дата следующего планового списания пересчитывается
    And счётчик попыток сбрасывается

  Scenario: Исчерпание попыток — пауза подписки
    Given выполнены все 3 попытки dunning без успеха
    Then подписка переходит в статус "PAST_DUE"
    And публикуется событие "subscription.payment_failed"
    And доступ к курсам приостанавливается по правилам lms

  Scenario: Отказ из-за протухшей карты
    Given рекуррентное списание отклонено с кодом "expired_card"
    Then повторы по тому же токену не планируются
    And студенту отправляется запрос на обновление карты (через lms)
```

## US-06 — Обработка чарджбэка

**Приоритет:** Should · **SP:** 8

> **Как** финансовый оператор,
> **я хочу** автоматически фиксировать чарджбэки и связывать их с подпиской,
> **чтобы** вовремя закрывать доступ и корректно вести учёт спорных операций.

### Acceptance Criteria

```gherkin
Feature: Обработка чарджбэков

  Scenario: Поступление чарджбэка по вебхуку
    Given по платежу "PAY-1" банк инициировал чарджбэк
    When PGS получает вебхук "chargeback.created" от эквайера
    Then создаётся запись чарджбэка со статусом "OPENED"
    And связанный платёж помечается флагом "disputed"
    And публикуется событие "payment.chargeback_opened" (триггер закрытия доступа)

  Scenario: Дедупликация повторного вебхука чарджбэка
    Given вебхук с external_event_id "EVT-CB-1" уже обработан
    When PGS повторно получает тот же вебхук
    Then состояние чарджбэка не меняется
    And PGS отвечает эквайеру HTTP 200

  Scenario: Решение по чарджбэку в пользу школы
    Given открытый чарджбэк по платежу "PAY-1"
    When эквайер присылает "chargeback.won"
    Then чарджбэк переходит в статус "WON"
    And флаг "disputed" с платежа снимается
```

## Нефункциональные требования (NFR)

| ID | Категория | Требование |
|------|-----------|------------|
| NFR-01 | Безопасность (PCI DSS) | PAN/CVV не проходят через PGS; токенизация на стороне эквайера (SAQ-A) |
| NFR-02 | Идемпотентность | Все мутирующие операции принимают `Idempotency-Key`; нет двойных списаний |
| NFR-03 | Производительность | p95 `POST /payments/charge` ≤ 900 мс при 300 RPS |
| NFR-04 | Доступность | SLA 99,95 %; недоступность ОФД не блокирует проведение платежа |
| NFR-05 | Отказоустойчивость | Circuit Breaker + Retry с backoff на всех внешних вызовах |
| NFR-06 | Согласованность | Доставка событий at-least-once через Transactional Outbox |
| NFR-07 | Комплаенс 54-ФЗ | Фискальный чек по каждому приходу/возврату ≤ 5 минут |
| NFR-08 | Наблюдаемость | Сквозной trace_id; дашборды success rate, latency, dunning recovery |
| NFR-09 | Безопасность вебхуков | Проверка HMAC-подписи входящих вебхуков эквайера/ОФД |
