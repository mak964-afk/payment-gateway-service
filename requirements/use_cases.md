# Use Case Specifications — Payment Gateway Service

Спецификации ключевых вариантов использования (формат Cockburn, расширенный):
основной (happy path), альтернативные и исключительные сценарии, пред-/постусловия.

## Акторы

| Актор | Тип | Роль |
|-------|-----|------|
| Студент | Первичный, человек | Оплачивает курс/подписку, отменяет подписку |
| `lms` (онлайн-школа) | Первичный, система | Инициирует оплату, холд/капчуру, возврат |
| Эквайер (PSP) | Вторичный, внешняя система | Проводит карточные операции, шлёт вебхуки |
| ОФД | Вторичный, внешняя система | Регистрирует фискальные чеки (54-ФЗ) |
| Планировщик | Первичный, система | Запускает рекуррент, dunning, автоотмену холда |
| Финансовый оператор | Вторичный, человек | Разбирает чарджбэки и инциденты фискализации |

## UC-01 — Провести оплату (charge)

| Поле | Значение |
|------|----------|
| **ID** | UC-01 |
| **Актор (первичный)** | `lms` (по инициативе Студента) |
| **Цель** | Принять оплату курса картой и зафискализировать чек |
| **Триггер** | Студент подтвердил оплату на стороне эквайера (получен токен) |
| **Предусловия** | Сумма и валюта валидны; передан `Idempotency-Key` |

### Основной сценарий (Main Success Scenario)

1. `lms` отправляет в PGS запрос оплаты (сумма, токен, состав чека, ключ идемпотентности).
2. PGS проверяет, не обрабатывался ли этот ключ ранее.
3. PGS сохраняет платёж в статусе `PENDING` (в одной транзакции с записью в outbox).
4. PGS вызывает адаптер эквайера для проведения операции.
5. Эквайер подтверждает списание.
6. PGS переводит платёж в `SUCCEEDED`, публикует `payment.succeeded`.
7. PGS асинхронно отправляет фискальный чек в ОФД (UC-04).

### Альтернативные сценарии

- **2a. Ключ идемпотентности уже использован:** PGS возвращает результат ранее
  проведённого платежа без побочных эффектов.
- **4a. Запрошен двухстадийный платёж (hold):** вместо списания выполняется
  авторизация; платёж переходит в `AUTHORIZED`, чек откладывается до capture (UC-03).

### Исключительные сценарии

- **5a. Эквайер отклонил операцию:** платёж переходит в `DECLINED`, причина
  возвращается `lms`; чек не формируется.
- **4b. Таймаут/обрыв связи с эквайером:** PGS не считает платёж успешным,
  запускает сверку статуса (reconciliation) по идемпотентному ключу, исключая
  двойное списание.

### Постусловия

- **Успех:** платёж `SUCCEEDED`, событие опубликовано, чек поставлен в очередь.
- **Неуспех:** платёж `DECLINED`/в сверке; деньги не списаны повторно.

## UC-02 — Рекуррентное списание по подписке

| Поле | Значение |
|------|----------|
| **ID** | UC-02 |
| **Актор (первичный)** | Планировщик |
| **Цель** | Списать оплату за новый период по сохранённому токену |
| **Триггер** | Наступила дата следующего списания активной подписки |
| **Предусловия** | Подписка `ACTIVE`; есть валидный токен карты |

### Основной сценарий

1. Планировщик инициирует списание по подписке.
2. PGS создаёт платёж `PENDING`, привязанный к подписке.
3. PGS вызывает эквайер с сохранённым токеном (без участия студента).
4. Эквайер подтверждает списание.
5. PGS переводит платёж в `SUCCEEDED`, сдвигает дату следующего списания.
6. PGS фискализирует чек (UC-04), публикует `subscription.renewed`.

### Альтернативные / исключительные сценарии

- **4a. Временный отказ (`try_again_later`/`insufficient_funds`):** запускается
  **dunning** (UC-05) — платёж `RETRY_SCHEDULED`, подписка остаётся `ACTIVE` в
  grace-периоде.
- **4b. Постоянный отказ (`expired_card`):** повторы не планируются; студенту
  уходит запрос на обновление карты; подписка → `PAST_DUE` по правилам lms.

### Постусловия

- **Успех:** период продлён, чек сформирован.
- **Отказ:** запущен dunning либо подписка приостановлена.

## UC-03 — Холдирование и списание (two-stage)

| Поле | Значение |
|------|----------|
| **ID** | UC-03 |
| **Актор (первичный)** | `lms` |
| **Цель** | Захолдировать сумму и списать её после подтверждения |
| **Триггер** | Запись на курс с двухстадийной оплатой |
| **Предусловия** | Получен токен; сумма валидна |

### Основной сценарий

1. `lms` инициирует авторизацию (hold) на сумму курса.
2. PGS вызывает эквайер для авторизации, платёж → `AUTHORIZED`.
3. Планировщик ставит таймер автоотмены холда (например, 7 дней).
4. После подтверждения `lms` инициирует capture.
5. PGS вызывает эквайер для списания, платёж → `SUCCEEDED`.
6. PGS фискализирует чек на фактически списанную сумму.

### Альтернативные / исключительные сценарии

- **4a. Частичный capture:** списывается часть суммы, остаток холда
  освобождается; чек — на списанную сумму.
- **3a. Истёк срок холда без capture:** планировщик отменяет холд у эквайера,
  платёж → `CANCELLED`, событие `payment.hold_expired`.
- **2a. Авторизация отклонена:** платёж → `DECLINED`, запись на курс не оплачена.

### Постусловия

- **Успех:** сумма списана, чек сформирован.
- **Отмена:** холд снят, деньги освобождены.

## UC-04 — Зафискализировать чек (54-ФЗ)

| Поле | Значение |
|------|----------|
| **ID** | UC-04 |
| **Актор (первичный)** | PGS (асинхронно) |
| **Цель** | Зарегистрировать фискальный чек в ОФД |
| **Триггер** | Платёж `SUCCEEDED` (приход) или проведён возврат |
| **Предусловия** | Известны состав чека, сумма, email/телефон студента |

### Основной сценарий

1. PGS формирует чек прихода (позиции, НДС, признак расчёта, контакт).
2. PGS отправляет чек в ОФД через адаптер.
3. ОФД регистрирует чек и возвращает фискальный признак (ФП) и номер.
4. PGS сохраняет чек в статусе `REGISTERED`, публикует `receipt.registered`.

### Альтернативные / исключительные сценарии

- **2a. ОФД недоступен (Circuit Breaker = OPEN):** чек → `PENDING_RETRY`, ставится
  команда повтора с backoff; платёж остаётся `SUCCEEDED` независимо от чека.
- **3a. ОФД отклонил чек (ошибка состава):** чек → `REJECTED`, эскалация
  Финансовому оператору для ручного исправления.
- **Триггер «возврат»:** формируется чек «возврат прихода» по UC-06.

### Постусловия

- **Успех:** чек `REGISTERED` ≤ 5 минут.
- **Отложено/неуспех:** чек в повторе или эскалирован; платёж не откатывается.

## UC-05 — Обработать вебхук эквайера

| Поле | Значение |
|------|----------|
| **ID** | UC-05 |
| **Актор (первичный)** | Эквайер (внешняя система) |
| **Цель** | Синхронизировать статус платежа/чарджбэка с фактическим |
| **Триггер** | HTTP-вебхук эквайера (оплата, возврат, чарджбэк) |
| **Предусловия** | Подпись вебхука валидна; платёж существует |

### Основной сценарий

1. Эквайер отправляет вебхук с `external_event_id` и типом события.
2. PGS проверяет HMAC-подпись запроса.
3. PGS дедуплицирует по `(provider, external_event_id)`.
4. PGS мапит внешний статус во внутренний (через ACL).
5. PGS проверяет допустимость перехода (конечный автомат) и обновляет состояние.
6. PGS пишет доменное событие в outbox, отвечает эквайеру `200`.

### Альтернативные / исключительные сценарии

- **2a. Подпись невалидна:** запрос отклоняется `401`, инцидент безопасности.
- **3a. Дубль события:** PGS отвечает `200` без изменения состояния.
- **5a. Недопустимый переход:** состояние не меняется, событие логируется, ответ `200`.
- **Тип `chargeback.created`:** открывается чарджбэк, платёж помечается `disputed`
  (US-06), публикуется `payment.chargeback_opened`.

### Постусловия

- **Успех:** состояние актуализировано, событие опубликовано.
- **Дубль:** состояние неизменно (идемпотентность).

## UC-06 — Оформить возврат (refund)

| Поле | Значение |
|------|----------|
| **ID** | UC-06 |
| **Актор (первичный)** | `lms` / Финансовый оператор |
| **Цель** | Вернуть студенту средства (полностью или частично) |
| **Триггер** | Запрос на возврат по успешному платежу |
| **Предусловия** | Платёж `SUCCEEDED`; сумма возврата ≤ остатка |

### Основной сценарий

1. `lms` инициирует возврат на сумму ≤ суммы платежа.
2. PGS вызывает эквайер для возврата.
3. Эквайер подтверждает возврат.
4. PGS переводит платёж в `REFUNDED` (или фиксирует частичный возврат).
5. PGS формирует фискальный чек «возврат прихода» (UC-04).
6. PGS публикует `payment.refunded` (триггер закрытия доступа в lms).

### Исключительные сценарии

- **2a. Эквайер недоступен:** возврат ставится в очередь повтора (Retry + CB).
- **1a. Сумма возврата превышает остаток:** запрос отклоняется `422`.

### Постусловия

- **Успех:** средства возвращены, чек возврата сформирован, событие опубликовано.
